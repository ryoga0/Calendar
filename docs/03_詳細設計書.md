## 4. API 詳細

### 4.1 認証

### POST `/auth/register`

新規ユーザーを作成し、そのままログイン状態にする。

リクエスト:

```json
{
  "email": "user@example.com",
  "password": "Password123",
  "user_name": "山田 太郎",
  "phone": "09012345678"
}
```

レスポンス:

```json
{
  "access_token": "jwt-token",
  "token_type": "bearer",
  "expires_in": 28800,
  "user": {
    "id": "user-uuid",
    "email": "user@example.com",
    "user_name": "山田 太郎",
    "phone": "09012345678"
  }
}
```

### POST `/auth/login`

メールアドレスとパスワードでログインする。

リクエスト:

```json
{
  "email": "user@example.com",
  "password": "Password123"
}
```

### 4.4 空き状況

### GET `/availability`

クエリ:

- `department_id`: 必須
- `date`: 必須。`YYYY-MM-DD`
- `exclude_appointment_id`: 任意。予約変更時に自分の予約を除外するために使う

レスポンス:

```json
{
  "department_id": "department-uuid",
  "date": "2026-04-10",
  "items": [
    {
      "time": "09:00",
      "start_at": "2026-04-10T09:00:00",
      "available": true,
      "reason": null
    },
    {
      "time": "10:00",
      "start_at": "2026-04-10T10:00:00",
      "available": false,
      "reason": "満員"
    }
  ]
}
```

### 空き状況生成ルール

- 平日: `09:00 / 10:00 / 11:00 / 13:00 / 14:00 / 15:00 / 16:00`
- 土曜: `09:00 / 10:00 / 11:00`
- 日曜: 空きなし
- 現在時刻以前の時間は `available = false`, `reason = "受付終了"`
- 既に予約済みの時間は `available = false`, `reason = "満員"`

---

### 4.5 予約

### GET `/appointments`

自分の予約一覧を返す。  
利用可能なクエリ:

- `from_date`
- `to_date`

ステータスが `confirmed` の予約だけを返す。

### GET `/appointments/{appointment_id}`

自分の予約詳細を返す。

### POST `/appointments`

リクエスト:

```json
{
  "department_id": "department-uuid",
  "start_at": "2026-04-10T09:00:00"
}
```

登録時の主な検証:

- 診療科が存在すること
- 診療科が有効であること
- 予約可能時間であること
- 同じ診療科に既存予約がないこと
- 同じ時間に他の予約がないこと

### PATCH `/appointments/{appointment_id}`

リクエスト:

```json
{
  "start_at": "2026-04-10T10:00:00"
}
```

変更時の主な検証:

- 自分の予約であること
- 予約が存在すること
- 新しい日時が予約可能であること
- 同じ日時を再指定していないこと
- 同じ時間に別の予約がないこと

### DELETE `/appointments/{appointment_id}`

対象予約を物理削除する。

レスポンス:

```json
{
  "status": "ok"
}
```

## 5. DB 設計

### 5.1 users

| 論理名 | 物理実装 | 型 | 備考 |
|--------|----------|----|------|
| id | `id` | String(36) | PK |
| email | `email` | String(255) | UNIQUE |
| password_hash | `password_hash` | String(255) | 必須 |
| user_name | `name` | String(100) | SQLAlchemy 属性は `user_name` |
| phone | `phone` | String(20) | 任意 |
| created_at | `created_at` | DateTime | 必須 |
| updated_at | `updated_at` | DateTime | 必須 |

補足:

- 既存 SQLite 互換のため、DB カラム名は `name`
- アプリケーション上は `user_name` に統一する

### 5.2 departments

| カラム | 型 | 備考 |
|--------|----|------|
| id | String(36) | PK |
| name | String(100) | 必須 |
| sort_order | Integer | 既定値 0 |
| is_active | Boolean | 既定値 true |

### 5.3 appointments

| カラム | 型 | 備考 |
|--------|----|------|
| id | String(36) | PK |
| user_id | String(36) | users.id 参照 |
| department_id | String(36) | departments.id 参照 |
| status | String(20) | 現在は `confirmed` を使用 |
| start_at | DateTime | 予約開始日時 |
| created_at | DateTime | 必須 |
| updated_at | DateTime | 必須 |

### 一意制約

SQLite では次の部分一意インデックスを作成している。

```sql
CREATE UNIQUE INDEX IF NOT EXISTS ux_appt_user_dept_confirmed
ON appointments (user_id, department_id)
WHERE status = 'confirmed'
```

これにより、同一利用者が同じ診療科に複数の有効予約を持つことを防ぐ。

## 7. エラー表示方針

- バックエンドは `code` と `message` を返す
- フロントは `message` を優先表示する
- HTTP ステータスだけでなく、利用者に必要な業務理由を表示する

代表例:

| code | message |
|------|---------|
| `NOT_FOUND` | 予約が見つかりません。 |
| `ALREADY_EXISTS` | この診療科はすでに予約済みです。変更する場合は予約一覧からお進みください。 |
| `TIME_CONFLICT` | 同じ時間に別の予約があります。別の時間をお選びください。 |
| `FULL` | この日時は満員です。別の日時をお選びください。 |
| `UNAVAILABLE` | この日時は予約できません。別の日時をお選びください。 |
| `INVALID_TIME` | 診療時間外のため、この日時は予約できません。 |
