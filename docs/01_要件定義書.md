## 1. 文書の目的

本書は、総合病院の患者が Web 画面から予約を行うための要件を定義する。  
現在の開発フェーズでは、**患者向け予約機能** を対象とし、FastAPI で動作検証を進める。  
Firebase 移行や管理者機能は、将来フェーズとして別途拡張する。

---

## 2. 対象範囲

### 2.1 現在の対象

- 患者の新規登録
- 患者のログイン
- プロフィール参照と更新
- 診療科一覧の表示
- 日付ごとの空き時間表示
- 予約の登録
- 予約一覧の表示
- 予約詳細の表示
- 予約日時の変更
- 予約のキャンセル

### 2.2 将来の拡張対象

- 管理者ログイン
- 診療科マスタ管理
- 休診日管理
- 通知機能
- Firebase への本番移行

### 2.3 対象外

- 電子カルテ連携
- 決済
- 保険請求
- 医師や看護師の勤務管理
- 電話予約との統合

## 4. 利用者像と UX 方針

このアプリは、高齢の利用者でも迷いにくいことを重視する。

- 文字は大きめにする
- ボタンは押しやすいサイズにする
- 1画面1目的に寄せる
- 戻る導線を分かりやすくする
- エラー理由を日本語で明確に表示する
- 専門用語や英語の内部用語をできるだけ見せない

対応デバイスは PC とスマートフォンを想定する。

## 6. 業務ルール

### 6.1 予約の単位

- 予約は **1時間単位** とする
- DB では `start_at` を保存し、`end_at` は `start_at + 1時間` として扱う

### 6.2 予約可能時間の扱い

- 永続化された `Slot` テーブルは持たない
- 予約可能時間は、診療時間ルールから **動的に計算** する
- 現在の FastAPI 実装では次の時間帯を使う

| 曜日 | 時間帯 |
|------|--------|
| 平日 | 09:00 / 10:00 / 11:00 / 13:00 / 14:00 / 15:00 / 16:00 |
| 土曜 | 09:00 / 10:00 / 11:00 |
| 日曜 | 受付なし |

### 6.3 予約制御

- 同じ利用者は、同じ診療科に対して同時に複数の有効予約を持てない
- 同じ利用者は、同じ時間に別の診療科の予約も持てない
- 過去日時には予約できない
- 無効化された診療科には予約できない
- 満員の時間には予約できない
- 予約変更では診療科は変えず、日時だけ変更する
- キャンセル時は DB から予約レコードを削除する

### 6.4 時刻の基準

- 日時判定は病院の基準タイムゾーンで行う
- 現在の設定値は `Asia/Tokyo`

## 7. データ要件

### 7.1 ユーザー情報

- ユーザーID
- メールアドレス
- パスワードハッシュ
- 名前
- 電話番号
- 作成日時
- 更新日時

### 7.2 予約情報

- 予約ID
- ユーザーID
- 診療科ID
- 予約開始日時
- 状態
- 作成日時
- 更新日時

### 7.3 診療科情報

- 診療科ID
- 診療科名
- 表示順
- 有効フラグ

## 8. 非機能要件

| 区分 | 要件 |
|------|------|
| 使いやすさ | 高齢の利用者でも操作しやすい見た目と導線にする |
| 可読性 | エラー理由を短く具体的に表示する |
| セキュリティ | ログイン必須ページは認証が必要。パスワードはハッシュ化して保存する |
| 保守性 | FastAPI から Firebase へ移行しやすい API 契約を意識する |
| 拡張性 | 管理者機能や休診日管理を追加できる構造にする |
| 性能 | 日付選択から空き時間表示まで、通常利用でストレスの少ない速度を目標とする |

## 10. 将来拡張で扱う項目

現時点では要件候補として扱い、現在の実装対象からは外している。

- 診療科追加・編集

- 予約件数上限ルール
