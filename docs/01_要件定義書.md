## 1. 文書の目的

本書は、総合病院の患者が Web 画面から予約を行うための要件を定義する。  
現在の運用前提は **React + Firebase** であり、患者向け予約機能を中心に扱う。

---

## 2. 対象範囲

### 2.1 現在の対象

- 患者の新規登録
- 患者のログイン
- プロフィール参照と更新
- 診療科一覧の表示
- 日付ごとの空き時間表示
- 予約の登録
- 予約一覧の表示
- 予約詳細の表示
- 予約日時の変更
- 予約のキャンセル

### 2.2 将来の拡張対象

- 管理者機能
- 休診日管理
- 通知
- Cloud Functions による予約処理の強化

### 2.3 対象外

- 電子カルテ連携
- 決済
- 保険請求
- 電話予約との統合

---

## 3. 利用者と UX 方針

利用者は 10 代から 80 代以上を想定する。  
高齢の利用者でも迷いにくい UI を重視する。

- 文字は大きめにする
- ボタンは押しやすいサイズにする
- 1 画面 1 目的に寄せる
- 戻る導線を分かりやすくする
- エラー理由を日本語で明確に表示する

対応デバイスは PC とスマートフォンを想定する。

## 5. 業務ルール

### 5.1 予約の単位

- 予約は 1 時間単位とする
- `end_at` は保存せず、`start_at + 1時間` として扱う

### 5.2 予約可能時間

- 永続化された `Slot` テーブルは持たない
- 予約可能時間は診療時間ルールから動的生成する

| 曜日 | 時間帯 |
|------|--------|
| 平日 | 09:00 / 10:00 / 11:00 / 13:00 / 14:00 / 15:00 / 16:00 |
| 土曜 | 09:00 / 10:00 / 11:00 |
| 日曜 | 受付なし |

### 5.3 予約制御

- 同じ利用者は、同じ診療科に対して同時に複数の有効予約を持てない
- 同じ利用者は、同じ時間に別の予約を持てない
- 過去日時には予約できない
- 無効化された診療科には予約できない
- 予約変更では診療科を変えず、日時だけ変更する
- キャンセル時は Firestore から予約を削除する

### 5.4 時刻の基準

- 日時判定は病院の基準タイムゾーン `Asia/Tokyo` を前提にする

## 6. データ要件

### 6.1 ユーザー情報

- ユーザー ID
- メールアドレス
- 名前
- 電話番号
- 作成日時
- 更新日時

### 6.2 予約情報

- 予約 ID
- ユーザー ID
- 診療科 ID
- 診療科名スナップショット
- 予約開始日時
- スロットキー
- 状態
- 作成日時
- 更新日時

### 6.3 診療科情報

- 診療科 ID
- 診療科名
- 表示順
- 有効フラグ

### 6.4 整合性維持用データ

- 同一診療科制御用ロック
- 同一時刻制御用ロック
- 診療科 × 時刻の満員判定用ロック

## 7. 非機能要件

| 区分 | 要件 |
|------|------|
| 使いやすさ | 高齢の利用者でも操作しやすい見た目と導線にする |
| 可読性 | エラー理由を短く具体的に表示する |
| セキュリティ | Firebase Authentication と Firestore Security Rules で本人以外のデータを扱えないようにする |
| 保守性 | 画面から Firebase 呼び出しを直接散らさず、モジュール境界を持つ |
| 拡張性 | 将来 Cloud Functions や管理機能を追加できる構造にする |
| 性能 | 日付選択から空き時間表示まで、通常利用で待ち時間を短く保つ |
