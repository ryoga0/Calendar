## 1. 文書の目的と範囲

### 1.1 目的

本書は、総合病院における**診療科別の外来予約**を、Webアプリ上で登録・変更・削除・閲覧できるようにするための、ビジネス要件およびシステム要件を定義する。

### 1.2 対象範囲

- **対象**: 予約を行う患者（利用者）向けの予約カレンダー機能
- **対象**: **院内運用者向け管理画面**（管理者ログイン、診療科マスタ、予約枠の運用、休診に伴う枠止め等）
- **対象外（本版）**: 院内カルテ連携、決済、保険請求、スタッフ向けシフト管理、SMS/電話予約との二重予約制御の詳細運用

### 1.3 前提・制約

- 技術は **React（フロントエンド）** と、まず **FastAPI（バックエンド・検証用）** で構築し、安定後に **Firebase**（認証・データストア等）へ移行する。
- 病院は**複数診療科**を持つ総合病院である。

## 3. 利用者像とUX方針

### 3.1 年齢層

10代から80代以上まで幅広い層が利用するため、次を要件とする。

- **視認性**: 十分なフォントサイズ、コントラスト、タップ領域の確保。
- **操作の単純化**: 画面遷移と用語を最小限にし、主要操作（予約する／変更／削除）を明示。
- **エラー表示**: 技術用語を避け、次に何をすべきかを一文で示す。
- **アクセシビリティ**: キーボード操作・スクリーンリーダー対応を基本設計以降で具体化（WCAG 2.1 レベルAAを目標）。

### 3.2 想定デバイス

PCブラウザおよびスマートフォン（レスポンシブ対応）。

## 4. 機能要件

### 4.1 機能一覧


| ID   | 機能名         | 概要                                                  |
| ---- | ----------- | --------------------------------------------------- |
| F-01 | ユーザー登録・ログイン | **メールアドレスとパスワードのみ**で本人確認とセッション／トークンによる利用制限。         |
| F-02 | プロフィール参照・編集 | 氏名、連絡先等のユーザー情報の閲覧・更新（任意項目は詳細設計で定義）。                 |
| F-03 | 予約登録        | 診療科・日時（空き枠）を選び予約を確定する。                              |
| F-04 | 予約一覧閲覧      | ログイン利用者の予約を日付・診療科で一覧表示する。                           |
| F-05 | 予約変更        | **同一診療科のまま、予約日時（枠）のみ**を変更する。                        |
| F-06 | 予約削除        | 利用者が予約を取り消す（キャンセル）。                                 |
| F-07 | 空き状況参照      | 診療科・日付を指定して予約可能枠を表示する（登録の前提）。                       |
| F-08 | 管理者ログイン     | 運用者のみが利用する認証（患者アカウントとは分離）。                          |
| F-09 | 診療科マスタ管理    | 診療科の**追加・編集・表示順・有効／無効**の維持。                         |
| F-10 | 予約枠の運用      | **1時間枠**の生成・参照・編集、および**枠単位の受付停止（枠止め）**。             |
| F-11 | 休診（枠止め）運用   | **診療科×日単位の休診**登録により、当該日の該当診療科の予約を受け付けない（枠を予約不可とする）。 |
| F-12 | 予約の監視・運用    | 予約一覧の検索・フィルタ、必要に応じた**代行キャンセル**等（監査ログを残す）。           |


### 4.2 診療フロー・同時予約に関する要件


| ID    | 要件                                                                                |
| ----- | --------------------------------------------------------------------------------- |
| BR-01 | **初診・再診の区別なく、予約の操作フローは同一**とする（画面や手順を分けない）。                                        |
| BR-02 | **同一診療科につき、確定（キャンセル以外）の予約は同時に1件まで**とする。                                           |
| BR-03 | **診療科が異なる**確定予約は、**合計で最大3件まで**同時に持てる（＝最大3診療科まで、各科1件ずつ）。                           |
| BR-04 | 予約の**時間枠は1時間単位**とする（例: 9:00～10:00、10:00～11:00）。システム上は各枠に開始時刻と終了時刻（開始から1時間後）を持たせる。 |


### 4.3 機能詳細

#### F-01 ユーザー登録・ログイン

- 認証方式は **メールアドレス＋パスワードのみ**とする（ソーシャルログイン、電話番号SMS認証等は用いない）。
- 新規利用者はアカウントを作成し、ログイン後にのみ予約操作が可能とする。
- パスワードポリシーおよび再設定フローは基本設計で定義する。

#### F-03 予約登録

- 同一利用者・同一枠の二重予約は不可とする。
- **申し込み先の診療科について、利用者に既に確定予約がある場合**は新規登録を拒否し、「この診療科では既に予約があります」等、分かりやすく表示する。
- **確定予約が既に3件ある場合**（＝異なる診療科に最大3件まで達した場合）は、**4件目以降の新規予約は不可**とし、理由を表示する。
- 枠が埋まった場合、**枠止め**または**休診日**の場合は登録不可とし、理由を表示する。

#### F-08～F-12 管理画面（概要）

- **F-08**: 管理者は患者用ログインとは別URL・別アカウントでログインする。
- **F-09**: 診療科名・並び順・有効フラグを画面から維持する。
- **F-10**: 枠の一括生成（例: 平日9–17時の1時間刻み）や個別枠の修正、**単一枠の受付停止**を行える。
- **F-11**: 休診日を診療科単位で登録し、患者向けの空き照会・予約APIでは当該日の枠を予約不可とする。
- **F-12**: 全患者の予約を条件検索し、運用理由があれば代行キャンセル等を行う（**誰がいつ操作したか**をログに残す）。

#### F-05 予約変更

- **診療科は変更できない。変更できるのは日時（別の予約枠）のみ**とする。
- 変更先の枠は**元の予約と同一診療科**に属すること。
- 変更可能な期限（例: 前日○時まで）は運用ルールとして定数または設定で持ち、詳細設計で実装仕様とする。
- 過去日時への変更は不可。

#### F-06 予約削除

- キャンセル後は一覧から状態が分かるようにする（論理削除またはステータス管理）。

## 5. データ要件

### 5.1 データ種別


| データ    | 主な項目（論理）                              | 備考                |
| ------ | ------------------------------------- | ----------------- |
| ユーザー情報 | 識別子、認証情報（メール・パスワードハッシュ）、プロフィール項目、作成日時 | 個人情報の取り扱いに注意。     |
| 予約情報   | 予約ID、利用者ID、診療科ID、予約枠ID、ステータス、作成・更新日時  | ステータス: 確定／キャンセル等。 |


### 5.2 マスタデータ（参照・運用）

- **診療科マスタ**: 診療科ID、名称、表示順、有効フラグ（**管理画面で編集**）。
- **予約枠**: 診療科ごとに **1時間刻み** の受付枠。枠ごとに**受付停止フラグ**（枠止め）を持てる。
- **休診カレンダー（論理）**: 診療科ID・日付単位で「休診」を表し、当該日の該当診療科は予約不可とする。

開発初期の初期データ投入には **DBシード**（付録A）を併用してよい。本番運用では管理画面を主とする。

## 6. 非機能要件


| 区分        | 要件                                                                                                              |
| --------- | --------------------------------------------------------------------------------------------------------------- |
| セキュリティ    | HTTPS、認証必須領域の分離、パスワードのハッシュ保存、個人情報の最小公開。**管理者と患者の認証・権限を分離**し、管理APIは管理者トークンのみ可とする。管理操作は**監査ログ**（操作者・日時・対象・内容）を残す。 |
| 性能        | 同時利用数十～数百規模を想定し、一覧・空き照会の応答は実用上許容範囲（目標: 主要画面3秒以内、詳細設計で数値化）。                                                      |
| 可用性       | Firebase 本番時はクラウド SLA に準拠。FastAPI 検証時は単一サーバ想定で可。                                                                |
| 保守性       | API を REST で整備し、バックエンド差し替え（FastAPI → Firebase Functions + Firestore 等）に耐える境界を設ける。                               |
| 法令・ガイドライン | 個人情報保護法に配慮した同意・利用目的の明示（別紙ポリシーと連携）。                                                                              |

## 7. 技術要件（要件レベル）


| 項目          | 内容                                                               |
| ----------- | ---------------------------------------------------------------- |
| フロントエンド     | React（**患者向けSPA** と **管理画面SPA** を同一リポジトリの `/admin` 等で分離する想定でもよい） |
| 認証（患者）      | **メールアドレス＋パスワード**（Firebase Authentication の Email/Password 等）    |
| 認証（管理者）     | **メールアドレス＋パスワード**（患者とは別テーブル／別Authユーザとし、権限を混在させない）                |
| バックエンド（第1段） | FastAPI（API の動作検証・結合テスト）                                         |
| バックエンド（第2段） | Firebase（Authentication、Firestore 等への移行）                         |
| データ保持       | ユーザー・予約の永続化。移行時はデータモデル互換を基本設計で検討。                                |

## 8. 用語定義


| 用語      | 定義                                                  |
| ------- | --------------------------------------------------- |
| 予約枠     | 診療科ごとに受け付ける **1時間単位** の時間帯（開始・終了を保持）。               |
| 利用者     | アプリに登録した患者（アカウント保有者）。                               |
| 総合病院    | 複数の臨床診療科を有する病院。                                     |
| DBシード   | 開発・検証用に、マスタ等の初期データをスクリプトやマイグレーションでDBへ一括投入すること（付録A）。 |
| 枠止め     | 管理者が**特定の予約枠**だけ受付を止めること（休診日とは別に、枠単位で運用）。           |
| 休診（日単位） | 管理者が**診療科×日**で登録し、その日の該当診療科を予約不可とすること。              |

## 9. 未確定事項・今後の判断

1. 予約変更・キャンセルの締切時刻および例外運用（電話対応との併用）。
2. Firebase 移行時のバッチ移行の要否（FastAPI 側に蓄積した検証データの扱い）。
3. 管理者の**多要素認証**・**IP制限**・アカウント発行フロー（院内方針）。
4. 監査ログの保存期間と参照権限。

## 付録A. DBシード（シードデータ）とは

**DBシード**とは、アプリケーションを初めて動かすときやテストするときに、**空のデータベースにあらかじめ入れておく初期データ**を、**コマンドやマイグレーションで自動的に流し込む**仕組み・作業のことである。

- **例**: 「内科・外科・整形外科」の診療科名、受付時間帯に沿った **1時間枠** のダミー予約枠、テスト用ユーザーなどを、SQL や JSON、ORM のシードスクリプトで一括INSERTする。
- **目的**: 手で管理画面から毎回入力しなくても、開発者・QA が同じ前提データで動作確認できるようにする。
- **本プロジェクトでの位置づけ**: 開発初期は **シードで診療科・枠を投入**しつつ、**管理画面**で休診・枠止め・マスタ修正を検証する。CIや新規環境立ち上げではシードで再現してもよい。
