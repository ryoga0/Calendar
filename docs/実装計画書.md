## 1. 前提

この計画は、現行コードと次の要件を基準にする。

- React と FastAPI で病院の予約カレンダーアプリを作る
- 総合病院で複数診療科がある
- 利用者は 10 代から 80 代以上
- 予約の登録、変更、削除、閲覧を行う
- まず FastAPI で検証し、その後 Firebase へ移行する

---

## 2. 現在の設計方針

- 永続化する `Slot` テーブルは持たない
- 予約可能時間は診療ルールから動的生成する
- 予約データは `appointments` に直接保持する
- 予約は `department_id + start_at` で扱う
- `end_at` は `start_at + 1時間` としてレスポンスで計算する
- 利用者名は `user_name` に統一する

## 4. フェーズごとの内容

### Phase 0: 現行コード整理

実施内容:

- `users` router を登録
- `display_name` を廃止し `user_name` に統一
- 未使用の `UserCreate` を削除
- `Slot` 前提コードを削除
- フロントで API エラー理由を表示できるよう整理
- ログイン失敗時の不要な登録画面遷移を廃止

完了条件:

- サーバが起動できる
- `Slot` 依存がなくなる
- 命名不整合が解消される

### Phase 1: FastAPI の患者向け API 完成

実施内容:

- 認証 API
- ユーザー API
- 診療科 API
- `availability` API
- 予約一覧、詳細、作成、変更、削除 API

完了条件:

- 患者の予約 CRUD が API で成立する
- 新規予約画面が `availability` API で組める

### Phase 2: React の患者向け UI 完成

実施内容:

- ID ベースのルーティング
- 認証ガード
- 公開トップ、ログイン、新規登録
- ホーム
- 新しい予約
- 予約一覧
- 予約詳細
- 予約変更
- プロフィール
- 戻るボタン
- 高齢者を意識した UI 改善

完了条件:

- 登録 → ログイン → 予約作成 → 一覧 → 変更 → 削除 まで通る

### Phase 3: テストと品質安定化

実施内容:

- `pytest` による API テスト追加
- `Vitest + Testing Library` によるフロント主要導線テスト追加
- README の詳細化
- 新しい予約カレンダーの見た目調整

完了条件:

- API テストが通る
- フロントテストが通る
- フロントのビルドが通る
